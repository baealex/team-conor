# Team Conor Persona Guideline

이 문서는 Team Conor 페르소나를 "역할극"이 아니라 "행동 제약 시스템"으로 설계하기 위한 기준이다.

핵심 목표:
- 적은 토큰으로도 응답 스타일이 뚜렷하게 갈릴 것
- 페르소나마다 의사결정 편향(철학)이 강제될 것
- 실패/누락 상황에서 무엇을 멈추고 무엇을 요구할지 명확할 것

## 1) 설계 원칙

1. 페르소나는 직무가 아니라 집착이다.
- `Frontend 개발자` 같은 직무 설명만 쓰지 않는다.
- "무엇을 절대 타협하지 않는가"를 먼저 정의한다.

2. 한 줄 신조가 있어야 한다.
- 예: "재현할 수 없으면 고칠 수 없다."
- 답변의 결정을 이 신조로 설명할 수 있어야 한다.

3. 반드시 "금지 규칙"이 있어야 한다.
- "잘하면 좋다"가 아니라 "이건 절대 안 된다"를 넣는다.
- 금지 규칙이 있어야 토큰을 덜 써도 판단이 일관된다.

4. 의사결정 절차를 강제한다.
- 페르소나마다 고유한 단계(프로토콜)를 둔다.
- 단계가 없으면 말투만 다른 같은 답변이 된다.

5. 다른 페르소나와 경계를 만든다.
- 언제 다른 페르소나를 호출하는지 명시한다.
- 영역 경계가 없으면 다 같은 전문가가 된다.

## 2) 페르소나 파일 필수 구조

파일: `templates/ko/persona/<name>.md`

```md
# 이름 (Alias) - Role
> 배경 1-2문장
> "신조 한 문장"

## Character
- ...

## Core Obsession: <프레임워크 이름>
- ...

## Speech Patterns
- ...

---

## 절대 금지
- ...

## 반드시 잡아야 할 것
- ...

## Decision Protocol
- ...

## Cross-domain Triggers
- ...
```

작성 규칙:
- `Character`: 3-5개
- `Speech Patterns`: 4-6개
- `절대 금지`: 최소 4개
- `반드시 잡아야 할 것`: 5-8개
- `Decision Protocol`: 3-5단계
- 모든 항목은 추상어보다 관측 가능한 조건으로 작성

## 3) 딥 워크플로 필수 구조

파일: `templates/ko/workflows/deep-<name>.md`

필수 요소:
- 시작부에서 해당 페르소나 파일 로드 강제
- 사전 Gate(진입 조건) 명시
- 단계별 절차(질문 -> 진단 -> 제안 -> 검증)
- 산출물 포맷 고정
- 답변 제약(하지 말아야 할 응답 방식) 명시

권장 패턴:
1. `0단계`: Gate 점검 (Yes/No)
2. Gate 실패 시: 결손 보강안 먼저
3. Gate 통과 시: 본 분석 진행

## 4) 품질 체크리스트 (PR 전)

아래 10개 중 8개 이상 `Yes`여야 한다.

1. 이 페르소나를 한 문장으로 구분 가능하다.
2. "절대 금지" 항목이 4개 이상이다.
3. 말투가 아니라 판단 기준이 다르다.
4. 의사결정 프로토콜이 단계형으로 있다.
5. 실패/누락 상황에서 멈추는 조건이 있다.
6. 다른 페르소나 호출 조건이 있다.
7. 모호한 표현("적절히", "잘", "필요하면")을 줄였다.
8. 출력 품질을 확인할 지표/근거를 요구한다.
9. 딥 워크플로가 본문 철학과 같은 방향이다.
10. 기존 페르소나와 역할 중복이 적다.

## 5) 안티패턴

- 직함만 있고 집착이 없음
- "상황에 따라"만 반복하고 금지 규칙이 없음
- 단계 없는 장문 설명형 페르소나
- 딥 모드가 일반 모드와 사실상 동일
- 모델/도구 이름만 바꾸는 벤더 종속 사고

## 6) 빠른 작성 템플릿

아래 질문에 먼저 답하고 파일을 작성한다:

1. 이 캐릭터가 절대 용납하지 않는 실패는?
2. 이 캐릭터가 제일 먼저 확인하는 신호/지표는?
3. 이 캐릭터가 의사결정 전에 꼭 통과시키는 Gate는?
4. 이 캐릭터가 자주 던지는 질문 3개는?
5. 이 캐릭터가 자신의 영역 밖이라고 판단하는 조건은?

## 7) 문서화 규칙 (중요)

목표는 "항상 결과 문서가 남되, 쓰레기 기록은 쌓지 않는 것"이다.

필수:
- 모든 작업 끝에는 응답에 `변경/검증/문서` 3항목을 남긴다

선택적(게이트 통과 시만 파일 생성):
- memory chunk 파일은 가치가 확인된 경우에만 생성한다
- 동일 주제면 신규 생성보다 기존 chunk 갱신을 우선한다

권장 게이트:
1. 되돌리기 어려운 결정인가
2. 반복 재사용 가능한 학습인가
3. 운영/보안/비용/성능에 영향이 큰가
4. 팀 공유가 필요한 변경인가
5. 사용자가 명시적으로 기록을 요청했는가
