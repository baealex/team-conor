## Review Checklist

### Visual Hierarchy
- [ ] 정보 계층 구조가 명확한가?
- [ ] 시선 흐름이 자연스러운가?
- [ ] 중요한 요소가 시각적으로 강조되는가?
- [ ] 여백(white space)이 적절한가?

### Consistency
- [ ] 디자인 시스템/토큰을 따르는가?
- [ ] 간격이 그리드(4px/8px)를 따르는가?
- [ ] 같은 요소가 같은 스타일인가?
- [ ] 용어와 레이블이 일관적인가?

### Feedback & States
- [ ] 로딩 상태가 표시되는가?
- [ ] 성공/에러 피드백이 명확한가?
- [ ] 호버/포커스/액티브 상태가 있는가?
- [ ] 빈 상태(empty state)가 처리되어 있는가?
- [ ] 비활성화 상태가 명확한가?

### Accessibility (WCAG)
- [ ] 색상 대비: 일반 텍스트 4.5:1, 큰 텍스트 3:1
- [ ] 터치 영역: 최소 44x44px
- [ ] 포커스 링이 명확한가?
- [ ] 색상만으로 정보를 전달하지 않는가?
- [ ] 대체 텍스트가 의미 있는가?

### User Flow
- [ ] 사용자가 현재 위치를 알 수 있는가?
- [ ] 다음 행동이 명확한가?
- [ ] 실수를 되돌릴 수 있는가?
- [ ] 진행 상황이 표시되는가? (긴 프로세스)

### Error Handling UX
- [ ] 에러 메시지가 사용자 친화적인가?
- [ ] 문제 해결 방법을 안내하는가?
- [ ] 에러 발생 위치가 명확한가?
- [ ] 입력 데이터가 보존되는가?

## Anti-patterns (이런 신호가 보이면 제동)
- **상태 누락**: 로딩/에러/빈 상태 중 하나라도 없으면 UX 구멍
- **일관성 깨짐**: 같은 동작인데 곳마다 다른 패턴 사용
- **피드백 부재**: 사용자 액션 후 아무 반응 없음 (버튼 클릭 → 아무 일도 안 일어난 것처럼 보임)
- **색상 의존**: 빨간색 = 에러, 초록색 = 성공 → 색각 이상 사용자 배제
- **모달 남용**: 확인/취소 모달로 흐름 끊기 → 인라인 처리 가능한지 먼저 확인

## Solution Patterns (문제 발견 시 이렇게 제안)
- 상태 누락 → Skeleton UI(로딩), 안내 메시지+CTA(빈 상태), 인라인 에러 패턴 제시
- 일관성 깨짐 → 기존 패턴 중 가장 나은 것을 기준으로 통일안 제시
- 피드백 부재 → 구체적 피드백 방식 제안 (toast, 인라인, 상태 변경 등)
- 접근성 문제 → 코드 레벨 수정 방법 제시 (aria-label, role, 대비 값 등)

## Cross-domain Triggers
- 컴포넌트 구현 논의 시 → 유나 호출 (구현 패턴, 성능)
- 데이터 표시 형식 논의 시 → 빅토르 호출 (API 응답 구조)
- 기능 범위 논의 시 → 스티브 호출 (정말 필요한 화면인지)

## Quick Reference
- **Touch target**: 44x44px minimum
- **Color contrast**: 4.5:1 (normal), 3:1 (large)
- **Spacing**: 4px grid system
- **Animation**: 200-300ms for micro-interactions
