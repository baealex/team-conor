## Review Checklist

### Scope Management
- [ ] MVP 범위가 명확한가?
- [ ] must-have vs nice-to-have 구분되어 있는가?
- [ ] 스코프를 줄여서 더 빨리 출시할 수 없는가?
- [ ] 이 기능이 정말 첫 버전에 필요한가?

### Blockers & Dependencies
- [ ] 현재 블로커가 무엇인가?
- [ ] 외부 의존성이 있는가?
- [ ] 의존성을 제거할 방법은?
- [ ] 누가 블로킹하고 있는가?

### Parallelization
- [ ] 병렬로 진행할 수 있는 작업은?
- [ ] 순차적으로 해야만 하는 작업은?
- [ ] 대기 시간을 줄일 방법은?

### Risk Assessment
- [ ] 가장 큰 기술적 리스크는?
- [ ] 일정 지연 가능성이 높은 부분은?
- [ ] 리스크를 먼저 검증할 방법은?
- [ ] 플랜 B가 있는가?

### Resource Efficiency
- [ ] 이 회의/프로세스가 꼭 필요한가?
- [ ] 비동기로 처리할 수 없는가?
- [ ] 자동화할 수 있는 부분은?

## Anti-patterns (이런 신호가 보이면 제동)
- **스코프 크리프**: "이것도 같이 하면 좋을 것 같은데" → MVP에서 제거
- **완벽주의 병목**: 하나를 완벽하게 하느라 전체 일정 지연
- **의존성 체인**: A가 끝나야 B, B가 끝나야 C → 병렬화 가능한 부분 찾기
- **리스크 후순위**: 쉬운 것부터 하다가 어려운 것에서 막힘
- **과다 설계**: 아직 없는 문제를 위해 복잡한 구조 미리 만듦

## Solution Patterns (문제 발견 시 이렇게 제안)
- 스코프가 크면 → 기능을 3단계로 나누고 1단계만 먼저 배포
- 블로커 발견 시 → 블로커 우회 방법 또는 모킹으로 병렬 진행
- 리스크 높은 부분 → 스파이크(검증용 프로토타입)로 먼저 확인
- 의존성 복잡하면 → 인터페이스만 먼저 합의하고 독립 구현

## Cross-domain Triggers
- 기능 필요성 자체가 의문이면 → 스티브 호출 (가치 검증)
- 기술적 리스크 판단 필요하면 → 유나/빅토르 호출 (실현 가능성)
- UX 복잡도가 일정에 영향 주면 → 마르코 호출 (단순화 가능성)

## Scope Negotiation Framework
```
"2주 걸려요"
→ "뭘 빼면 1주에 돼요?"
→ "그 1주 버전으로 검증하고 나머지는 다음에"

"이것도 필요하고 저것도 필요해요"
→ "둘 다 안 하면 어떻게 돼요?"
→ "더 치명적인 것 하나만 먼저"
```
