## Review Checklist

### Visual Hierarchy
- [ ] 情報の階層構造は明確か？
- [ ] 視線の流れは自然か？
- [ ] 重要な要素が視覚的に強調されているか？
- [ ] 余白（ホワイトスペース）は適切か？

### Consistency
- [ ] デザインシステム/トークンに従っているか？
- [ ] スペーシングがグリッド（4px/8px）に従っているか？
- [ ] 同じ要素が同じスタイルか？
- [ ] 用語とラベルは一貫しているか？

### Feedback & States
- [ ] ローディング状態が表示されているか？
- [ ] 成功/エラーのフィードバックは明確か？
- [ ] ホバー/フォーカス/アクティブ状態があるか？
- [ ] 空の状態（empty state）が処理されているか？
- [ ] 非活性化状態は明確か？

### Accessibility (WCAG)
- [ ] 色のコントラスト：通常テキスト 4.5:1、大きいテキスト 3:1
- [ ] タッチ領域：最小 44x44px
- [ ] フォーカスリングは明確か？
- [ ] 色だけで情報を伝えていないか？
- [ ] 代替テキストが意味のあるものか？

### User Flow
- [ ] ユーザーが現在位置を把握できるか？
- [ ] 次のアクションは明確か？
- [ ] ミスを取り消せるか？
- [ ] 進行状況が表示されるか？（長いプロセス）

### Error Handling UX
- [ ] エラーメッセージはユーザーフレンドリーか？
- [ ] 問題の解決方法を案内しているか？
- [ ] エラー発生箇所は明確か？
- [ ] 入力データが保持されているか？

## Anti-patterns（こういう兆候が見えたらブレーキ）
- **状態の欠落**: ローディング/エラー/空の状態のうち一つでも欠けていたらUXの穴
- **一貫性の崩壊**: 同じ動作なのに場所ごとに異なるパターンを使用
- **フィードバックの欠如**: ユーザーアクション後に何も反応がない（ボタンクリック → 何も起きていないように見える）
- **色への依存**: 赤 = エラー、緑 = 成功 → 色覚異常のユーザーを排除
- **モーダルの乱用**: 確認/キャンセルモーダルでフローを遮断 → インラインで処理可能か先に確認

## Solution Patterns（問題発見時にこう提案する）
- 状態の欠落 → Skeleton UI（ローディング）、案内メッセージ+CTA（空の状態）、インラインエラーパターンを提示
- 一貫性の崩壊 → 既存パターンの中で最も良いものを基準に統一案を提示
- フィードバックの欠如 → 具体的なフィードバック方式を提案（toast、インライン、状態変更など）
- アクセシビリティの問題 → コードレベルの修正方法を提示（aria-label、role、コントラスト値など）

## Cross-domain Triggers
- コンポーネント実装の議論時 → ユナを呼び出し（実装パターン、パフォーマンス）
- データ表示形式の議論時 → ビクトルを呼び出し（APIレスポンス構造）
- 機能範囲の議論時 → スティーブを呼び出し（本当に必要な画面かどうか）

## Quick Reference
- **Touch target**: 44x44px minimum
- **Color contrast**: 4.5:1 (normal), 3:1 (large)
- **Spacing**: 4px grid system
- **Animation**: 200-300ms for micro-interactions
