# Team Conor

AIコーディングアシスタント（Claude）のためのチームペルソナシステムです。

## なぜ必要なのか？

一人で開発するとコードレビューがありません。AIにレビューを頼んでも、「クリーンコードを書きましょう」「エラーハンドリングを追加しましょう」のような抽象的なアドバイスしか返ってきません。

実際のチームでは違います。PMは「これMVPに必要？」と聞き、デザイナーは「ローディング状態がないけど？」と指摘し、バックエンド開発者は「N+1クエリじゃない？」とキャッチしてくれます。それぞれの専門領域から**具体的な問題**を**具体的な解決策**とともに提示します。

Team Conorはこの体験をAIコーディングアシスタントに組み込みます。5人の専門家ペルソナがそれぞれのチェックリストとアンチパターンカタログでコードをレビューし、問題発見時には解決パターンまで提案します。

## インストール

```bash
npx team-conor
```

### CLIオプション

```bash
# 対話型セットアップ（デフォルト）
npx team-conor init

# 言語選択（ko, en, ja）
npx team-conor init --lang ja

# 非対話型モード（CI/CDなどインタラクション不可の環境）
npx team-conor init --name "田中太郎" --no-interaction

# 既存ファイルを強制上書き
npx team-conor init --name "田中太郎" -y

# ヘルプ
npx team-conor --help
npx team-conor init --help

# バージョン確認
npx team-conor --version
```

## 生成されるファイル

```
CLAUDE.md                    # AI設定ファイル
.conor/
├── persona/                 # チームメンバーペルソナ
│   ├── user.md              # ユーザー（プロジェクトオーナー）
│   ├── planner.md           # スティーブ（プロダクト戦略）
│   ├── pm.md                # エロン（実行PM）
│   ├── designer.md          # マルコ（UX）
│   ├── frontend.md          # ユナ（Frontend）
│   └── backend.md           # ヴィクトル（Backend）
└── memory/                  # プロジェクトコンテキスト
    ├── summary.md           # コア要約（CLAUDE.mdから参照）
    ├── project.md           # 技術スタック、アーキテクチャ
    ├── decisions.md         # 主要な意思決定記録
    └── learnings.md         # 学んだパターン、バグ修正
```

| 名前 | 役割 | 背景 | 視点 |
|------|------|------|------|

| エロン | 実行PM | SpaceX初期メンバー | 「いつ終わる？」- スコープ調整、ブロッカー除去、MVP集中 |

| ユナ | Frontend | Chrome ブラウザ初期開発チーム出身、TC39参加者 | 「リレンダリング何回？」- パフォーマンス、型安全性、ブラウザネイティブAPI |

## 使い方

### チームメンバーの呼び出し

```
ユナ、このコンポーネントをレビューして
ヴィクトル、API設計をチェックして
マルコ、このUIのアクセシビリティは大丈夫？
```

### チームディスカッション

```
会議しよう、この機能をどう実装するか
```

### コード作業

```
このバグを修正して  → 実際の作業を実行後、関連ペルソナの視点から自己検証
```

## 主な機能

### チェックリストベースのレビュー

各ペルソナは抽象的なアドバイスの代わりに、具体的なチェックリストでフィードバックします：

```
ユナ：このコンポーネントを見たんだけど...
- [ ] useEffectの依存配列に`user`が抜けています
- [ ] この状態はサーバーコンポーネントに引き上げられます
- [x] 型はちゃんとできていますね
```

### アンチパターン検出

各ペルソナは自分の領域のアンチパターンを自動的に検出します：

```
ヴィクトル：これN+1クエリじゃないですか？ループの中でDBを呼び出していますね。
→ JOINかバッチクエリに置き換えるべきです。
```

### 解決パターンの提案

問題を指摘する際には必ず解決方向を一緒に提示します（診断だけで処方のないフィードバックは禁止）：

```
マルコ：ローディング状態がないので、ユーザーはフリーズしたと思うでしょう。
→ Skeleton UIパターンを適用すれば解決します。[具体的なコードを提示]
```

### クロスドメイン協業

関連領域を発見した場合、他のペルソナを自動的に呼び出します：

```
スティーブ：この機能の範囲が広がってきているんだけど...
→ エロン、スコープ管理を見てくれる？

エロン：MVPで何を削れば半分の時間でできるか見てみます。
```

### Memoryシステム

プロジェクトコンテキストを`.conor/memory/`に記録し、セッション間の連続性を維持します：

- **summary.md**：コアコンテキスト要約（常に参照される）
- **project.md**：技術スタック、アーキテクチャ、コンベンション
- **decisions.md**：主要な技術的意思決定と根拠
- **learnings.md**：発見したパターン、バグ修正

### アップデートサポート

すでに設定済みのプロジェクトで`npx team-conor`を再実行すると：
- 既存ファイルとの差分をdiff表示
- 上書き / スキップ / バックアップを選択可能
- **memoryファイルは自動的に保存**（ユーザーデータの保護）

## カスタマイズ

各ペルソナファイル（`.conor/persona/*.md`）を編集して：
- チェックリスト項目の追加・修正
- アンチパターン、解決パターンをプロジェクトに合わせて調整
- プロジェクトの技術スタックを反映
- チームコンベンションの追加
